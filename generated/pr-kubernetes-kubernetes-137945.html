Kubelet DRA: Rollback Prepared Drivers on Partial NodePrepareResources Failure

This PR fixes a bug in the kubelet's Dynamic Resource Allocation (DRA) logic where, if a Pod's resource preparation partially fails across multiple drivers, already-prepared resources are not rolled back. This previously led to stranded device capacity and Pods stuck in a failed state. The PR introduces a mechanism to track and rollback successful preparations when any driver in the batch fails.

Key Changes

  • Introduces tracking of successfully prepared DRA drivers during NodePrepareResources.
  • Implements rollback logic to call NodeUnprepareResources on all previously successful drivers if any driver fails.
  • Ensures rollback is triggered on various error conditions, including unknown claim UIDs, per-claim errors, skipped claims, and local cache write failures.
  • Prevents resource leaks and stuck Pods by guaranteeing cleanup of partially prepared resources.
  • Adds regression test to verify correct rollback behavior in multi-driver scenarios.

File Changes

  • pkg/kubelet/cm/dra/manager.go — modified
    Core DRA resource preparation logic (controller/worker): adds preparedByDriver tracking, rollbackPrepared function, and integrates rollback calls into error paths in NodePrepareResources.
  • pkg/kubelet/cm/dra/manager_test.go — modified
    Unit/regression tests: adds TestPrepareResourcesRollbackOnPartialFailure to validate rollback behavior when one of multiple drivers fails during resource preparation.

Risks & Impact

  • Rollback logic may fail silently if NodeUnprepareResources encounters errors, potentially masking cleanup issues.
  • Changes affect the resource preparation path for all Pods using DRA with multiple drivers, increasing the blast radius.
  • Potential for new edge cases if driver implementations do not handle repeated prepare/unprepare calls idempotently.
  • If rollback is incomplete, resource leaks or stuck Pod states could still occur.

Testing Notes

  • Added targeted regression test for partial failure and rollback scenario.
  • Test asserts correct error propagation, exact rollback calls, and cache state.
  • Manual/automated tests should cover multi-driver, single-driver, and error-path cases.
  • Oncall should monitor for unexpected device leaks or Pods stuck in FailedPrepareDynamicResources after rollout.
  • Rollback errors are logged for observability but not propagated, so log monitoring is important.
